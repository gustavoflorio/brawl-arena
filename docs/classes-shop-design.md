# Design: Classes + Shop + Currency

Generated by /office-hours on 2026-04-24
Branch: main
Repo: brawl-arena
Status: APPROVED
Mode: Builder

---

## Problem Statement

Hoje, todo player no Brawl Arena tem o **mesmo moveset** (Jab1→Jab2→Jab3 + Heavy + DodgeRoll + DI), hardcoded em `Constants.Combat.Moves`. Falta variação de gameplay e razão pra progressão fora do rank ladder.

Queremos introduzir **classes de personagem** — começando com 3 (Boxer, Taekwon, Ballerina) — e uma **loja in-game** com **currency ganha jogando** que permite desbloquear classes adicionais. As formas concretas de obtenção de currency (kill rewards, daily quests, level-up bonuses, etc.) ficam **fora do escopo desta entrega** — só persistimos o saldo. Implementadas em PR posterior.

---

## What Makes This Cool

- **Identidade do player.** Hoje você é "um boneco do Brawl Arena". Depois desta entrega você é "o cara que joga de Ballerina" — começa a haver assinatura, mains, gente que prefere uma classe sobre outra.
- **Loop de progressão fora do rank.** Jogou → ganhou currency → desbloqueou classe → quer testar → jogou mais. É uma alavanca de retenção independente do MMR.
- **Infra extensível.** Depois desta entrega, adicionar a 4ª classe (Wrestler? Karate? Capoeira?) custa **um arquivo novo** e zero refactor. O sistema vira fundação pra anos de adições.
- **Pragmatismo do Roblox.** Sem mecânicas únicas (sem stance/charge/guard exóticos). As 3 classes nascem mecanicamente idênticas — diferenciação de gameplay vem em PRs futuras quando você troca os movesets de Taekwon e Ballerina, classe a classe.

---

## Constraints

- **Plataforma:** Roblox / Luau `--!strict`. Rojo. Sem dependências externas.
- **Arquitetura existente:** Server fonte única de verdade; Services seguem `Init(deps) → Start()`; ServiceLoader carrega ordenado; 2 Remotes (`BrawlRequest`, `BrawlState`); ProfileService (módulo já vendorizado em `Server/Modules/`).
- **Persistência:** Profile schema versionado (`RankSchemaVersion` já existe como precedente; vamos seguir o mesmo padrão pra schema de classes).
- **Anti-cheat:** validação **toda no server** — cliente nunca decide se compra foi feita ou se classe foi desbloqueada.
- **DESIGN.md:** UI da loja precisa respeitar tokens de cor/tipografia/spacing existentes. Sem motion de tela. Motion permitida só contida em widgets.

---

## Premises

Confirmadas com o usuário antes da geração do doc:

1. **Classes começam mecanicamente idênticas** — todas 3 usam `{ Jab1, Jab2, Jab3, Heavy }` clonado de `Constants.Combat.Moves`. Trocar movesets reais é trabalho futuro, classe a classe.
2. **Boxer é default free; Taekwon e Ballerina são compráveis** — onboarding zero-friction.
3. **Single currency in-game** — uma moeda só (proposta: `BrawlCoins`), persistida no profile. Formas de ganho fora desta entrega.
4. **Classe = string no profile** — schema ganha `Currency`, `UnlockedClasses` (set), `EquippedClass` (string). Class data mora em módulos.
5. **CombatService lê moveset da classe equipada do attacker** — substitui acesso direto a `Constants.Combat.Moves` por lookup via classe.
6. **Loja é client UI + server-validated transaction** — zero confiança no cliente; lock por player durante transação.

## Approaches Considered

### Approach A — Extensão mínima (bolt-on em Constants)
- `Constants.Classes = { Boxer={...}, Taekwon={...}, Ballerina={...} }` inline.
- BrawlRequest ganha actions `BuyClass` e `EquipClass`.
- ShopUI hardcoded com as 3 classes.
- **Pros:** menor diff, zero arquivos novos.
- **Cons:** Constants.lua incha; adicionar classe nova exige mexer em config central; mistura combate + economy.

### Approach B — Module-per-class + ShopService dedicado ⭐ (escolhida)
- Cada classe é um ModuleScript em `ReplicatedStorage/Shared/Classes/`.
- `ShopService.lua` novo encapsula compra/equip/catalog.
- `BrawlShop` é Remote separado (RemoteFunction + RemoteEvent).
- Profile schema +3 campos.
- **Pros:** classe nova = 1 arquivo; domínios separados (combate vs shop); module-first é amigo do `--!strict`.
- **Cons:** mais arquivos novos; outro Remote no Net.

### Approach C — Catalog-as-Instance (Roblox-native)
- `ReplicatedStorage/ClassCatalog/{Boxer,Taekwon,Ballerina}/` como folder tree.
- Cada classe = Folder com Moves (ModuleScript), DisplayName (StringValue), Price (IntValue), Icon (Decal).
- **Pros:** editável no Studio Explorer; replicação nativa.
- **Cons:** mistura code+data; pior pra `--!strict` (Instance:FindFirstChild → `any`); git diffs mais frágeis via Rojo.

---

## Recommended Approach

**Approach B — Module-per-class + ShopService dedicado.**

### Rationale
- Escala bem: a 4ª/5ª classe custa **1 arquivo**.
- `--!strict` é trivial em ModuleScripts; em Instance trees vira batalha.
- Separação de domínios (`CombatService` vs `ShopService`) já é o padrão atual.
- O custo extra (mais arquivos no diff inicial, +1 Remote) compra capacidade de evolução por anos.

### ClassDefinition (contrato do módulo)

```lua
--!strict
-- ReplicatedStorage/Shared/Classes/Boxer.lua

local ReplicatedStorage = game:GetService("ReplicatedStorage")
local Constants = require(ReplicatedStorage.Shared.Constants)

local Boxer = {
    Id = "Boxer",                       -- chave canônica (nunca muda)
    DisplayName = "Boxer",
    Description = "Pressão constante. Jabs rápidos, sem mistério.",
    IconAssetId = "rbxassetid://...",   -- placeholder até ter arte
    Price = 0,                          -- free
    IsDefault = true,                   -- equipada para players novos
    Moves = Constants.Combat.Moves,     -- placeholder: clone exato dos moves atuais
                                         -- (PR futura: substitui por moveset próprio)
}

return Boxer
```

Taekwon e Ballerina seguem o mesmo formato com `IsDefault = false`, `Price = 500`, e `Moves = Constants.Combat.Moves` por enquanto. Quando você for caprichar nos movesets reais classe a classe (PR futura), substitui essa linha.

### Classes/init.lua (registry)

```lua
--!strict
-- ReplicatedStorage/Shared/Classes/init.lua

local script_ = script :: Instance

type ClassDefinition = {
    Id: string,
    DisplayName: string,
    Description: string,
    IconAssetId: string,
    Price: number,
    IsDefault: boolean,
    Moves: { [string]: any },  -- shape espelha Constants.Combat.Moves
}

local Classes = {}
local catalog: { [string]: ClassDefinition } = {}
local defaultId: string? = nil

for _, child in ipairs(script_:GetChildren()) do
    if child:IsA("ModuleScript") and child.Name ~= "init" then
        local def = require(child) :: ClassDefinition
        assert(def.Id == child.Name, `Class module name mismatch: {child.Name}`)
        catalog[def.Id] = def
        if def.IsDefault then
            assert(defaultId == nil, `Multiple default classes`)
            defaultId = def.Id
        end
    end
end

assert(defaultId, "No default class found")

function Classes.GetClass(id: string): ClassDefinition?
    return catalog[id]
end

function Classes.GetDefault(): ClassDefinition
    return catalog[defaultId :: string]
end

function Classes.GetCatalog(): { ClassDefinition }
    local list = {}
    for _, def in pairs(catalog) do
        table.insert(list, def)
    end
    return list
end

return Classes
```

### Profile schema delta

```lua
-- PlayerDataService.lua: PROFILE_TEMPLATE ganha:

local PROFILE_TEMPLATE: Profile = {
    -- ... campos existentes ...
    Currency = 0,
    UnlockedClasses = { Boxer = true },  -- always present; default unlocked
    EquippedClass = "Boxer",
    ClassSchemaVersion = 0,              -- bump para 1 na primeira migração
}
```

Migration roda no load (paralelo ao `migrateRankSchema`):

```lua
local CURRENT_CLASS_SCHEMA = 1

local function migrateClassSchema(data)
    if (data.ClassSchemaVersion or 0) >= CURRENT_CLASS_SCHEMA then
        return
    end
    -- Migration 0 → 1: profiles antigos não tinham classes. Garante
    -- que Boxer está unlocked + equipped + currency=0.
    data.Currency = data.Currency or 0
    data.UnlockedClasses = data.UnlockedClasses or {}
    data.UnlockedClasses.Boxer = true
    if not data.EquippedClass or not data.UnlockedClasses[data.EquippedClass] then
        data.EquippedClass = "Boxer"
    end
    data.ClassSchemaVersion = CURRENT_CLASS_SCHEMA
end
```

### ShopService API (server)

```lua
-- ServerScriptService/Server/Services/ShopService.lua

type BuyResult = {
    success: boolean,
    reason: string?,    -- "AlreadyOwned" | "InsufficientFunds" | "UnknownClass" | "InTransaction"
    newBalance: number?,
}

function ShopService:BuyClass(player: Player, classId: string): BuyResult
function ShopService:EquipClass(player: Player, classId: string): boolean
function ShopService:GetCatalogFor(player: Player): {
    classes: { ClassCatalogEntry },  -- inclui owned/equipped flags por player
    balance: number,
}
```

**Lock por player durante transação:** flag em memória `_inTransaction[player] = true` — outras BuyClass durante a janela retornam `InTransaction`. Liberada após profile commit.

### CombatService delta

Lookup atual:
```lua
local move = Constants.Combat.Moves[moveKey]
```

Vira:
```lua
local profile = self._services.PlayerData:GetProfile(attacker)
local classDef = Classes.GetClass(profile.Data.EquippedClass) or Classes.GetDefault()
local move = classDef.Moves[moveKey]
```

Fallback pra default em caso de classe equipada inválida (defesa em profundidade — não deveria acontecer com a migration certa).

### Remote: BrawlShop

`Net/Remotes.lua` ganha:

```lua
Remotes.Shop = {
    Folder = "Remotes",
    Function = "BrawlShop",  -- RemoteFunction (request/response)
    Event = "BrawlShopEvent", -- RemoteEvent (server→client push, ex: catalog refresh)
}
```

Cliente chama `BrawlShop:InvokeServer({ action = "Buy", classId = "Taekwon" })` etc. Server discrimina por `action`.

**Por que RemoteFunction e não RemoteEvent?** Compras precisam de resposta síncrona (UI tem que mostrar erro/sucesso). Estado de partida (BrawlState) é fire-and-forget; loja é request/response.

### ShopUI (alto nível)

- Modal full-screen ou sliding panel ancorado.
- Lista das 3 classes em cards (Display Name, Description, Icon, Price).
- Cards têm 3 estados visuais: **Equipped** (badge verde), **Owned** (botão "Equipar"), **Locked** (botão "Comprar — N coins").
- Insufficient funds: botão dimmed + tooltip "Faltam X coins".
- Transação ack: card pulsa `semantic.success` (do DESIGN.md) por 200ms; saldo HUD atualiza.
- Erro: shake horizontal contido no card (motion permitida — widget-level), label vermelha (`semantic.error`).
- Trigger: botão "Loja" no lobby (não acessível durante arena).
- Tipografia / cores: tokens do DESIGN.md (`bg.surface`, `text.primary`, `GothamBold` pra labels, `Sarpanch` pra display de preço).

## Open Questions

- **Preço de Taekwon e Ballerina:** sugestão inicial 500 coins cada (placeholder). Calibrar quando definir taxa de ganho de currency.
- **Equipar durante match?** Proposta: **não** — só permitir EquipClass quando `PlayerState == InLobby`. Trocar de classe no meio da arena confunde state, animations, e abre brecha de exploit. Confirmar.
- **Saldo inicial:** Currency começa em 0. Confirmar se deve ter starter pack (ex: 50 coins de welcome) ou se onboarding é 100% Boxer-puro até primeiro ganho.
- **Persistência cross-server:** ProfileService já garante isso (load/release pattern). Sem trabalho adicional.
- **UI de "primeiro ganho de classe":** quando o player compra Taekwon ou Ballerina, vale uma celebração (banner/sound)? Fora de escopo desta entrega — flagar pra polish posterior.
- **Análise:** AnalyticsService deve registrar eventos `class_purchased`, `class_equipped`. Adicionar a `Constants.Analytics.Events`.

## Success Criteria

Esta entrega está **done** quando:

1. Player novo entra → spawneia Boxer equipada → consegue lutar com moveset atual sem regressão.
2. Player abre a loja no lobby → vê 3 cards (Boxer Equipped, Taekwon Locked 500, Ballerina Locked 500) + saldo (0 inicialmente).
3. Player com saldo suficiente clica "Comprar Taekwon" → server valida → currency desce → classe vai pra UnlockedClasses → card vira Owned.
4. Player com Taekwon Owned clica "Equipar" → EquippedClass muda → próxima arena spawneia com Taekwon (mesmo moveset, mas sistema sabe).
5. Profile sobrevive a logout/login: classes desbloqueadas e classe equipada persistem.
6. Tentar comprar com currency insuficiente → server retorna `InsufficientFunds` → UI mostra erro sem crash.
7. Tentar comprar classe já owned → server retorna `AlreadyOwned`.
8. Race condition: 2 cliques rápidos em "Buy" → segundo retorna `InTransaction` ou `AlreadyOwned`, nunca dupla cobrança.
9. ServerOps via Studio: setar `Currency = 9999` no profile manualmente → reflete no client após reload.
10. Type checking `--!strict` passa em todos os arquivos novos/modificados.

## Implementation Order (sugerido)

1. **Schema migration** — PlayerDataService template + migrateClassSchema. Test: profile antigo carrega, profile novo nasce com defaults.
2. **Classes module + 3 classe modules** — todas idênticas, todas apontam pra `Constants.Combat.Moves`. Test: `Classes.GetCatalog()` retorna 3.
3. **CombatService refactor** — lookup via `Classes.GetClass(profile.EquippedClass)`. Test: Boxer-only player luta sem regressão.
4. **ShopService** — Buy, Equip, GetCatalogFor com lock e validação. Test: unit-test by hand no Studio (RemoteFunction:InvokeServer com mocks).
5. **Net/Remotes** — adicionar BrawlShop. Auto-criação no `Main.server.lua` (segue padrão dos remotes existentes).
6. **ShopController + ShopUI** — UI mínima viável (cards estáticos, botão fechar, integração com saldo).
7. **AnalyticsService** — adicionar `class_purchased` e `class_equipped` events.
8. **QA e ship** — `/qa` com 2+ players no Studio.

Cada step é commitável de forma independente. Steps 1-3 podem ir num PR (zero impacto visível). Steps 4-7 viram outro PR (feature loja).

## What I noticed about how you think

- **Pragmatismo.** Sua resposta "sem mecânicas únicas, é um jogo de Roblox" cortou meses de over-engineering. Você sabe que feature creep mata projetos solo. Isso é raro.
- **Identifica o pivô certo.** Quando eu propus 3 esboços de moveset com mecânicas únicas (guard, charge, twirl), você não rejeitou as ideias — você rejeitou **o timing**. "Por hora vamos fazer todos iguais, depois eu troco" é a frase de um eng que entende que infra vem antes de polish, e que entrega incremental ganha de big bang.
- **Vocabulário sólido.** "Pressão / burst / mobility" não é jargão random — é o tripé clássico de fighting game design. Você conhece esse domínio.
- **Cuida de edge cases sem ser perguntado.** Você nem precisou que eu mencionasse "e se a classe equipada não existir mais?" — sua intuição de schema versionado (`RankSchemaVersion` já está no código) mostra que você pensa em migração desde antes.
